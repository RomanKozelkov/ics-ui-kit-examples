# Промо-календарь: сравнение двух реализаций

Сравнение двух прототипов таймлайна промо-календаря для выбора базы под дальнейшее развитие.

- `src/examples/promo-calendar-dnd` — на основе библиотеки **`dnd-timeline`** (`@dnd-kit/core` под капотом).
- `src/examples/promo-calendar-rct` — **полностью своя реализация** (несмотря на имя «rct»).

---

## ⚠️ Важное уточнение по именам

Папка `promo-calendar-rct` названа в честь `manually-build-timeline`, но **эту библиотеку не использует**. Поиск по импортам даёт **ноль** упоминаний `manually-build-timeline` в коде. Зависимость висит в `package.json` мёртвым весом.

Фактически `rct` — это **ручной рендер**: абсолютное позиционирование `left = dayIndex * dayWidth`, без какой-либо таймлайн-библиотеки.

**Поэтому реальный выбор не «dnd-timeline vs manually-build-timeline», а:**

| Вариант | Что это |
|---|---|
| **dnd** | Библиотека `dnd-timeline` — range в миллисекундах, `valueToPixels`, хуки `useItem` / `useRow`, drag/resize/zoom встроены |
| **rct** | Свой код — пиксели как `индекс дня × ширина дня`, ноль таймлайн-либ |

---

## Что общего (идентичный код в обоих)

Архитектура всего, кроме движка позиционирования, **совпадает дословно**:

- `buildGroupTree` — рекурсивная многоуровневая группировка (N уровней)
- `assignLanes` — greedy interval partitioning, «выше та, что раньше»
- `GroupSection` — рекурсивный рендер секций + сворачивание через `collapsedPaths: Set<string>`
- Двухуровневая шапка: месяцы (level 1) + дни (level 2), адаптивная плотность (`showDayNumbers` по порогу ширины)
- Палитра на 6–8 цветов (`colorForIndex`)
- Тултип (Наименование / Длительность / Бренд / Канал / SKU)
- Обрезка промо по краям года (`overflowLeft` / `overflowRight`)
- Sticky левая колонка при горизонтальном скролле
- Today-линия

Разница **только** в способе позиционирования полосок и наличии drag.

## Ключевое различие движка

| Аспект | dnd-timeline | rct (свой) |
|---|---|---|
| Модель координат | range в ms → `valueToPixels()` | `dayIndex * dayWidth` |
| Источник позиции | хуки либы (`useItem`, `useRow`) | прямой расчёт в `prepareItems` |
| Drag & drop | **встроен** (`@dnd-kit`) | отсутствует |
| Resize | **встроен** (выключен флагом) | отсутствует |
| Zoom / pan шкалы | **встроен** (через range) | отсутствует (статичный `dayWidth`) |
| Реактивность | пересчёт пикселей на каждый range-change | статичные px, мемоизация |

## Сравнение по критериям

| # | Критерий | dnd-timeline | rct (свой) | Победитель |
|---|---|---|---|---|
| 1 | Двухуровневая шапка | Есть, через `valueToPixels`, завязано на range/zoom | Есть, голый flex `dayCount × dayWidth`; 3-й уровень = ещё один flex-ряд | **rct** (проще расширять) |
| 2 | Много элементов | Виртуализации нет из коробки; либа headless → подключаешь `react-virtual` сам (есть офиц. пример) | Виртуализации нет; но добавить проще — рендер целиком твой | **rct** (по простоте), оба решаемы |
| 3 | Многоуровневая группировка | `buildGroupTree`, N уровней | **тот же код** | ничья (готово) |
| 4 | Сворачивание групп | `collapsedPaths` Set | **тот же код** | ничья (готово) |
| 5 | Кастомный бейдж | Контент бейджа — твой React внутри обёртки `useItem`. CSS дублировать не надо | Бейдж — твой React, absolute. CSS дублировать не надо | оба: **отдавай свой React**, не дублируй CSS |
| 6 | Клики по элементам | Конфликт click vs drag → нужен activation constraint (distance, ~5 строк) | `onClick` на div, ноль конфликтов | **rct** |
| 7 | Добавить/удалить 1 элемент | state → пересчёт + апдейт контекста либы; ок, но тяжелее | state → memo-пересчёт → reconcile; легче | **rct** (чуть легче), оба без лагов на единичной операции |
| 8 | Будущие нужды (см. ниже) | Закрывает редактирование/зум из коробки | Всё интерактивное — писать с нуля | **dnd** |

## Разбор вопросов по `dnd-timeline` (v3.1.1, проверено по node_modules)

**Есть ли drag & drop?**
Да — это ядро библиотеки, построена на `@dnd-kit/core`. В прототипе `promo-calendar-dnd` уже работает: перетаскивание промо меняет даты через state `overrides`. Resize тоже встроен, сейчас отключён одним флагом.

**Умеет ли виртуализацию?**
Сама — нет. Библиотека **headless**: отдаёт хуки + позиционные стили (`valueToPixels`, `itemStyle`, `rowStyle`), а строки рендеришь ты. README прямо рекомендует связку с `react-virtual`. У автора есть официальный Virtual-пример на `@tanstack/react-virtual`. Виртуализация — твоя задача прикрутить, библиотека не мешает.

**Плоха ли на 10k элементов?**
Без виртуализации — ляжет (десятки тысяч DOM-нод). Это **не вина библиотеки** — `rct` без виртуализации ляжет так же. С `react-virtual` — это штатный путь dnd-timeline для больших данных, работает нормально.

**Подвох виртуализации (для обоих вариантов):** структура «группы + lanes + сворачивание» даёт переменную высоту строк + drag через границу виртуального окна. Нетривиально. В `rct` встроить проще (рендер полностью твой), в `dnd` — официально поддержано, но добавляется борьба с drag-сенсорами на виртуальных рядах.

## 🏁 Финальный вердикт

### Рекомендация: **`dnd-timeline`**

**Обоснование:** главное подтверждённое будущее требование — **редактирование промо (drag/resize)**. `dnd-timeline` закрывает его из коробки, плюс бесплатно даёт зум/пан шкалы. Писать перенос дат и resize вручную поверх `rct` — это много работы, которую библиотека уже сделала.

**Цена решения (что доделать):**
1. Click-vs-drag: добавить activation distance constraint (~5 строк), чтобы клик не ловился как драг.
2. Включить resize: убрать `resizeHandleWidth: 0`, повесить `onResizeEnd`.
3. Виртуализация для больших данных: воткнуть `@tanstack/react-virtual` (есть офиц. пример).

**Когда выбор был бы в пользу `rct`:** если бы фича оставалась чисто дисплейной (display-only), `rct` выигрывал по простоте, контролю и лёгкости виртуализации — 5 из 7 механических критериев. Но редактирование меняет картину.

### Что делать с `rct`

Не выбрасывать — оставить как **референс лёгкого рендера**. Логика `assignLanes` и `buildGroupTree` в нём идентична dnd-версии; при прикручивании виртуализации lane-раскладку можно переиспользовать. Также `rct` — готовый запасной вариант, если интеграция drag+virtual в dnd окажется дороже ожидаемого.

### Уборка

Если фиксируем `dnd-timeline` — удалить мёртвую зависимость `manually-build-timeline` из `package.json` (нигде не используется).
