# Промо-календарь: `react-calendar-timeline` vs `dnd-timeline`

Сравнение двух прототипов под `task.md` для выбора базы.

- `src/examples/promo-calendar-react-calendar-timeline` — библиотека **`react-calendar-timeline`** (`0.30.0-beta.18`). Готовый таймлайн-компонент.
- `src/examples/promo-calendar-dnd` — библиотека **`dnd-timeline`** (`3.1.1`, headless, на `@dnd-kit/core`). Каркас рендеришь сам.

> ⚠️ Это **другое** сравнение, чем в `PROMO_CALENDAR_COMPARISON.md` (там был ручной рендер «rct» vs dnd). Здесь — настоящая библиотека `react-calendar-timeline` против `dnd-timeline`.

---

## TL;DR

| | react-calendar-timeline | dnd-timeline |
|---|---|---|
| Тип | Готовый компонент (batteries-included) | Headless (хуки + позиционные стили) |
| Время до рабочего дисплея | **Быстро** — шапка/стек/маркеры из коробки | Дольше — header/grid/lanes пишешь сам |
| Многоуровневая группировка | ❌ модель групп **плоская**, N уровней не поддерживает | ✅ свой `buildGroupTree`, N уровней |
| Стекинг непересекающихся полос | ✅ `stackItems` из коробки | ❌ свой `assignLanes` (написан) |
| Drag / resize / zoom | ⚠️ есть, но это не наш кейс (выключено) | ✅ ядро библиотеки |
| Стилизация | CSS-переопределения `.rct-*` обязательны | Чистый Tailwind, ноль чужого CSS |
| Зрелость версии | **beta** (`0.30.0-beta.18`) | stable `3.1.1` |
| Качество TS-типов | Частично; часть типов **не экспортируется** | Чистые, всё экспортится |
| Знание ИИ | Высокое (старая популярная либа) | Среднее (но `@dnd-kit` под капотом ИИ знает) |

**Развилка:** если календарь — это **дисплей с многоуровневой группировкой** → `dnd-timeline` (RCT не умеет вложенные группы). Если группировка одноуровневая и нужно **быстро** → `react-calendar-timeline` даёт больше из коробки.

## Насколько легко реализовать `task.md`

`task.md` = чистый дисплей: двухуровневая шапка, адаптивная плотность дней, группировка+сворачивание, полоски с тултипом, стек непересекающихся, обрезка по краям года, today-линия. Drag/resize в `task.md` **нет**.

| Требование `task.md` | RCT | dnd |
|---|---|---|
| Двухуровневая шапка (месяц/день) | ✅ `<DateHeader unit="primaryHeader">` + `<DateHeader unit="day">` — из коробки | ⚠️ свой `TimelineHeader` через `valueToPixels` (написан) |
| Адаптивная плотность дней (засечки/номера) | ✅ `intervalRenderer` + `interval.labelWidth` порог | ✅ свой расчёт `dayPx >= DAY_NUMBER_MIN_PX` |
| Выходные/границы недель заливкой | ✅ `verticalLineClassNamesForTime` → CSS-класс | ✅ свой класс на дне |
| Today-линия | ✅ `<TodayMarker>` из коробки | ⚠️ свой div (тривиально) |
| Стек «выше та, что раньше» | ✅✅ **`stackItems` — даром**, не пишешь lane-логику | ❌ свой `assignLanes` (greedy, написан) |
| Группировка по полю | ⚠️ только **один** уровень (плоский `groups[]`) | ✅ N уровней |
| Сворачивание секций | ⚠️ руками: фильтр `items` + `height` свёрнутой группы | ✅ `collapsedPaths: Set`, рекурсивный рендер |
| Полоска: имя + `/ дней`, ellipsis | ✅ `itemRenderer` (свой React) | ✅ свой React в `PromoItem` |
| Тултип (5 полей) | ✅ ICS `Tooltip` внутри `itemRenderer` | ✅ то же |
| Обрезка по краям года | ✅ клампишь `start_time/end_time` в `useTimelineData` | ✅ то же в `useGroupedRows` |
| Sticky левая колонка | ✅ `sidebarWidth` встроен | ⚠️ свой sticky (`position: sticky left-0`) |

**Вывод по `task.md`:** на **одноуровневой** группировке RCT закрывает больше пунктов из коробки (стек, шапка, маркеры, сайдбар) — меньше своего кода. Но как только нужна **вложенная** группировка (Канал → Бренд → …), RCT упирается: его модель `groups` плоская, вложенность отображается только эмуляцией (флатить дерево в плоский список + отступы + ручное сворачивание). dnd с самого начала работает с деревом.

## По фичам

### Удобство работы с типами
- **dnd-timeline:** ✅ чистые экспортируемые типы — `Range`, `OnRangeChanged`, `Span`, хуки типизированы дженериками. `Modifier` берётся из `@dnd-kit`. Код типобезопасен без костылей.
- **RCT:** ⚠️ типы есть (`TimelineKeys`, `IntervalRenderer`, `ReactCalendarGroupRendererProps`), но **`ItemRendererProps` наружу не экспортируется** — в `PromoItemRenderer.tsx` пришлось вручную пере-объявить тип (см. коммент в коде). Дженерики `Timeline<TItem, TGroup>` работают, но кастомные поля item-а приходится протаскивать самому.

### Дока
- **RCT:** обширная (старая популярная либа), много примеров/issues/Stack Overflow. **Но:** актуальная ветка `0.30.x` — **beta**, доки частично отстают от 0.30 API; ряд гайдов под старый стабильный `0.28`.
- **dnd-timeline:** дока компактнее, есть офиц. примеры (в т.ч. virtual на `@tanstack/react-virtual`). Меньше комьюнити, но `@dnd-kit` под капотом — гигантская дока и тонна примеров.

### Понимает ли ИИ
- **RCT:** ✅ высоко — много кода в обучении (популярная либа годами). Риск: ИИ помнит **старый** `0.28` API, а стоит `0.30-beta` (другие пропсы) → путаница версий.
- **dnd-timeline:** ⚠️ средне как сама либа, но ✅ высоко по `@dnd-kit`. Паттерн «хуки + valueToPixels» ИИ схватывает.

### Сколько допиливать самому
- **RCT:** мало для одноуровневого дисплея (либа делает каркас). Много — для **вложенной** группировки и тонкой стилизации (борьба с `.rct-*`).
- **dnd-timeline:** много изначально (весь каркас твой), но всё уже написано в прототипе и полностью под контролем. Потолок гибкости выше.

## Контрольные вопросы

### 1) Двухуровневая шапка
- **RCT:** ✅ из коробки — два `<DateHeader>` (`primaryHeader` = месяцы, `day` = дни). **Третий уровень** (напр. недели) — добавляешь ещё `<DateHeader unit="...">`, но кастомные единицы ограничены набором либы.
- **dnd:** ⚠️ свой `TimelineHeader` (два абсолютных ряда через `valueToPixels`). Любой N-й уровень — просто ещё один flex-ряд. Гибче, но пишешь сам.

### 2) Много элементов / виртуализация из коробки
- **RCT:** ❌ **виртуализации нет**. Рендерит все items/строки разом. На 10k+ — тормоза. Своей точки расширения под виртуализацию практически нет (компонент не headless) → прикрутить сложно.
- **dnd:** ❌ из коробки нет, **но** headless → официально поддержан `@tanstack/react-virtual` (есть пример). Подвох: переменная высота строк (группы+lanes+сворачивание) + drag через границу окна — нетривиально, но решаемо. Под виртуализацию dnd **гибче**.

### 3) Многоуровневая группировка
- **RCT:** ❌ **не поддерживает**. `groups` — плоский массив. В текущем прототипе сделан **только один уровень** (`groupBy: GroupField | null`). Вложенность = ручная эмуляция (флатить дерево + отступы + самому прятать строки при сворачивании).
- **dnd:** ✅ `buildGroupTree` рекурсивный, N уровней, `GroupSection` рекурсивно рендерит. Это **главное преимущество** dnd под наш кейс.

### 4) Сворачивание групп
- **RCT:** ⚠️ нет нативного — эмуляция: убрать items группы + ужать `height` строки группы. Работает для одного уровня; для вложенных — усложняется.
- **dnd:** ✅ `collapsedPaths: Set<string>` + рекурсия. Любой уровень сворачивается, дети скрываются автоматически.

### 5) Стилизация — CSS или Tailwind, кастомный рендеринг
- **RCT:** ⚠️ **придётся писать CSS.** Каркас либы стилизуется только переопределением `.rct-*` классов (`styles.css` уже это делает: `.rct-sidebar`, `.rct-dateHeader`, `.rct-item` с `!important` и т.д.). Контент полосок/групп — **свой React** через `itemRenderer`/`groupRenderer` (там Tailwind свободно). Итого: гибрид CSS-переопределений (каркас) + Tailwind (контент).
- **dnd:** ✅ **чистый Tailwind, чужого CSS ноль.** Либа отдаёт только позиционные inline-стили (`itemStyle`, `rowStyle`, `style` контейнера). Всю визуалку рисуешь сам. Кастомный рендер — by design (каждый элемент твой).

### 6) Клики по элементам
- **RCT:** ✅ `onItemClick`/`onItemSelect` из коробки; при `canMove={false} canSelect={false}` конфликтов drag/click нет.
- **dnd:** ⚠️ click vs drag конфликтует — нужен activation constraint (`@dnd-kit` distance/delay, ~5 строк), иначе клик ловится драгом. В прототипе drag активен, так что обязательно.

### 7) Добавить / удалить 1 элемент — насколько тяжело
- **RCT:** ✅ легко — меняешь массив `items`, либа сама reconcile-ит. Пересчёт `useTimelineData` (memo) дешёвый. **Но** `stackItems` пере-раскладывает все полоски группы (своё, но O(n) на группу).
- **dnd:** ✅ легко — меняешь `state` → `useGroupedRows` (memo) → пере-`assignLanes` затронутой группы → reconcile. Единичная операция без лагов.
- Обе: единичный add/remove дёшев. На больших данных без виртуализации обе просядут одинаково (DOM-ноды).

## Зрелость / риски

| | RCT | dnd |
|---|---|---|
| Версия | `0.30.0-beta.18` — **beta**, API может меняться | `3.1.1` — stable |
| React 19 | ✅ peer `^18 \|\| ^19` | ✅ |
| Доп. peer-зависимости | `interactjs`, `dayjs` | нет (тянет `@dnd-kit`) |
| Главный риск | beta-нестабильность + плоская модель групп | весь каркас на тебе |

## Вердикт (по требованиям, без учёта моей реализации)

- **Нужна многоуровневая группировка** (Канал→Бренд и т.п.) → **`dnd-timeline`**. RCT её архитектурно не тянет — только эмуляция, которая съест выигрыш «из коробки».
- **Группировка одноуровневая, важна скорость старта, drag не нужен** → **`react-calendar-timeline`** даёт стек/шапку/маркеры/сайдбар даром, меньше своего кода. Цена: CSS-переопределения, beta, слабее типы.
- **В будущем редактирование (drag/resize/zoom)** → **`dnd-timeline`** (ядро либы).

По совокупности `task.md` (где группировка многоуровневая — Канал/Клиент/Бренд, со сворачиванием секций) перевес у **`dnd-timeline`**: единственный из двух, кто нативно держит вложенные группы и полную стилизацию на Tailwind. `react-calendar-timeline` сильнее только в одноуровневом сценарии за счёт `stackItems`/шапки из коробки.
